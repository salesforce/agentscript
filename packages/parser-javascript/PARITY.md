## 1. Both-valid divergences (3 cases)

These are inputs where both parsers produce error-free trees, but the s-expressions differ. They are cosmetic — the parsed structure is semantically equivalent. All three remaining cases involve where a trailing comment attaches near a block boundary; they are locked in by `test/parity-gap-regressions.test.ts`.

### 1a. Comment attachment at a block boundary

A comment trailing a mapping attaches to a different parent in each parser. Both are valid CST representations.

**Example** (DIVERGE #1):

```
topic test:
   rasoning_actions:
       go_complete: @utils.transition to @topic.complete
           available when @variable.c#ount >= 10
```

- parser-js: `(comment)` is a sibling of the outer `reasoning_actions` mapping (one level shallower)
- tree-sitter: `(comment)` sits inside the innermost `available when` mapping (one level deeper)

### 1b. Comment-only line after a template

A comment-only line following a template block scalar is retained by parser-js but dropped by tree-sitter.

**Example** (DIVERGE #2 — bare `\r` + null byte in template context):

```
topic foo:
    reasoning:
        instructions: ->
            | \0hello
        \r     # commented
```

- parser-js: keeps the trailing `(comment)` as a sibling of the top mapping
- tree-sitter: omits the comment entirely

### 1c. Trailing block after a dash-line template

A bare identifier line trailing a dash-line template is kept as an empty mapping element by parser-js.

**Example** (DIVERGE #3):

```
config:
    message: |\
       ]-*hello {!@user.name}
        - go\rodbye
```

- parser-js: emits a trailing empty `(mapping_element key: (key (id)))` for the dedented `- go...odbye` line
- tree-sitter: absorbs it, producing no trailing element

> **Note:** these three cases all involve mutation-only control characters (bare `\r`, null bytes) or ambiguous indentation where "correct" attachment is debatable. See the header comment in `test/parity-gap-regressions.test.ts` for why they remain unfixed.

## 2. Error disagreements (160 cases)

One parser reports an error and the other does not. Split: **54 js-error/ts-valid**, **106 js-valid/ts-error**. The per-bucket counts below are approximate — they are derived by pattern-matching the mutated inputs (SEED=42) and a single input can fall into more than one bucket.

### 2a. parser-js has error, tree-sitter accepts (54 cases)

#### DEDENT-breaks on comments (primary cause, ~30 cases)

parser-js unconditionally breaks block loops on DEDENT. When a comment appears at reduced indentation inside a block, the DEDENT causes parser-js to exit the block early, wrapping remaining content in ERROR nodes. tree-sitter's GLR parser peeks past the indent disruption.

**Example** (DISAGREE #8):

```
system:
   config:
 #     settgs:
         timeout: "30@s"
```

- parser-js: exits `config:` block on DEDENT before `#`, puts `timeout` in ERROR
- tree-sitter: attaches comment to `config:` and continues parsing `timeout`

Three parser locations affected:

- `parseMapping()` — DEDENT/EOF break
- `parseProcedure()` — loop guard and inner break on DEDENT

**Example** (DISAGREE #19 — comment at wrong indent in procedure):

```
topictest:
   instructions: ->
     # First comment
      # Second comment
      run @action.test
```

- parser-js: `# First comment` triggers DEDENT, wrapping `# Second comment` in ERROR
- tree-sitter: both comments parsed normally

#### With-statement / run-statement recovery (~6 cases)

When a `with` clause is at wrong indentation or the `run` target is malformed, parser-js fails to associate the `with` with the `run`.

**Example** (DISAGREE #5):

```
topic test:
   reasoning_actions:
       capture_info: @utils.setVariables
            description: "Capture member information"
           with member_number=@variables.member_number
           with member_email=@variables.member_email
```

- parser-js: second `with` goes into ERROR
- tree-sitter: both `with` statements parsed

#### Mid-token newlines breaking block structure (~7 cases)

A newline inserted mid-identifier breaks the block key, causing parser-js to see two separate blocks.

**Example** (DISAGREE #4):

```
topic test:
   reasonin_actions:
       mixed_logic: @actions.
check
           available when ...
```

- parser-js: `@actions.\ncheck` breaks — `check` starts a new block with MISSING `:`
- tree-sitter: recovers and continues the original mapping

#### Miscellaneous (~7 cases)

- Colon separated from key by newline (`info\n  : 3 + 33`)
- Comment replacing identifier prefix (`#varaiables:\n   char: ...`)
- Tab inside identifier (`this_is_a_very_long_varia\tle_name`)

### 2b. parser-js accepts, tree-sitter has error (106 cases)

#### Bare `\r` inside identifiers (~19 cases)

parser-js treats bare `\r` (not followed by `\n`) as whitespace. tree-sitter rejects it with ERROR.

**Example** (DISAGREE #2):

```
topic test:
   rea\roning_actions:
       ...
```

- parser-js: `\r` is whitespace, key becomes `rea oning_actions`
- tree-sitter: ERROR node around the `\r`

#### Null bytes in strings/comments (~6 cases)

parser-js terminates the string at `\0` and continues; tree-sitter emits `UNEXPECTED '\0'`.

**Example** (DISAGREE #22):

```
      # Int\0rmediate step
```

- parser-js: parses comment normally (stops at newline)
- tree-sitter: `(ERROR (UNEXPECTED '\0') ...)`

#### Truncated keywords (`ru`, `un`, `rn` instead of `run`) (~15 cases)

parser-js has more lenient error recovery — it treats the invalid token as an expression and continues. tree-sitter wraps the entire statement block in ERROR.

**Example** (DISAGREE #25):

```
      if @var.x:
         run @action.yes
      else:
         un @action.no
```

- parser-js: `un` parsed as expression, `@action.no` parsed separately
- tree-sitter: entire source wrapped in ERROR

#### Space before colon in block key (~5 cases)

parser-js tolerates `config9 :` (space before colon); tree-sitter requires colon immediately after key.

**Example** (DISAGREE #7):

```
config9 :
   agent_name: "Test!i!ng agent nam-e"
```

- parser-js: valid parse
- tree-sitter: ERROR on the space-colon pattern

#### Tab inside identifiers (~13 cases)

parser-js treats tab as whitespace (splitting identifier into two key parts); tree-sitter treats it as an error.

**Example** (DISAGREE #9):

```
config:
   h\tst: "localhost"
```

- parser-js: key becomes `(key (id) (id))` — split into two identifiers
- tree-sitter: ERROR around the tab

#### Template scope differences (~10 cases)

parser-js considers content after a dedented line as still part of the template. tree-sitter ends the template at the dedent and wraps trailing content in ERROR.

**Example** (DISAGREE #44):

```
key twowords: |
   template text solely on new line
.underindented
     this one is more indented
```

- parser-js: entire content as one template
- tree-sitter: template ends, remaining in ERROR

#### Line continuation handling (~3 cases)

parser-js ignores `\` at end of line (backslash before newline); tree-sitter treats it as `line_continuation` token and may error.

**Example** (DISAGREE #27):

```
      if @var.outer:\
         if @var.inner:
```

- parser-js: ignores `\`, parses as two if statements
- tree-sitter: `(ERROR (line_continuation) ...)` inside the if

#### Escape sequences in strings (~5 cases)

When mutations create sequences like `\r` inside string content, parser-js may interpret them as whitespace while tree-sitter rejects them or interprets differently.

#### Malformed operators (~2 cases)

parser-js is lenient with `|==` or similar — it may parse the `|` as template start and recover. tree-sitter wraps in ERROR.

## 4. How to reproduce

All commands run from `packages/parser-javascript/`.

```bash
# Static parity — 281 corpus inputs, both parsers must agree
npx vitest run test/parity.test.ts

# Fuzz parity — 5620 mutations, tracks disagreements as metrics
npx vitest run test/fuzz-parity.test.ts

# Verbose: see every DISAGREE and DIVERGE case with inputs and s-expressions
FUZZ_VERBOSE=1 npx vitest run test/fuzz-parity.test.ts

# Single-parser fuzz — parser-js invariants only (no crashes, source preserved, valid CST)
npx vitest run test/fuzz.test.ts

# Error recovery CST coverage comparison
npx vitest run test/error-recovery.test.ts

# Reproduce a specific fuzz failure (change SEED value)
SEED=42 npx vitest run test/fuzz-parity.test.ts
```

### Environment requirements

- tree-sitter native bindings must be installed (`pnpm install` from repo root)
- Fuzz-parity and error-recovery tests skip gracefully if tree-sitter is unavailable
- Fixed seed (42) ensures reproducibility across runs

## 5. Open gaps

### Residual DEDENT-on-comment breaks (~30 cases)

parser-js still exits a block early when a comment lands at an ambiguous indent inside it (§2a), wrapping following content in ERROR where tree-sitter recovers. The comment-attachment work in #200/#266 closed the bulk of these; the remainder are mutation-heavy edge cases.

### Control-character handling (majority of §2b, 106 cases)

Most disagreements are js-valid/ts-error splits from mutation-only control characters: bare `\r` (~19), tabs inside identifiers (~13), null bytes (~6), line continuations (~3). parser-js's lexer treats these as whitespace; tree-sitter's scanner rejects them. Whether to tighten parser-js or loosen the grammar is an open design question — none of these patterns occur in valid authored `.agent` files.
